EC2 remains one of the most critical AWS cloud services currently available!!!

"Elastic Compute Cloud" popularly known as "EC2", is an AWS cloud services that lets you rent a virtual 
computer(called an "instance") and perform actions on it â€” just like you would on your own computer, but in the cloud.

> AWS EC2 simply has to do with creating/provisioning and managing servers/virtual machines
> in the cloud. With an AWS account, you simply create the virtual machine, get access to it from AWS, then connect 
> to it remotely and use it as intended.

Some EC2 use-cases include:

1. Hosting a website.

2. Deploying API servers.

3. Running databases.

4. Training machine learning models.

5. Setting up CI/CD servers.

More...

> The next couple of articles on the Zed Labs blog, are cloud and DevOps contents that will be quite 
> heavy on using AWS EC2. Hence I though to create this stand-alone guide, to share how to create a new AWS EC2 
> instance and gain access into it for performing tasks on the cloud.

In this piece:

1. You'll learn how to create/provision a new AWS EC2 instance.
2. You'll get to discover the different ways I recommend to access and use your EC2 instance.

Let's Dive in.

## Creating the EC2 Instance.

- Log into your AWS account, and select EC2 from the list of available services.

- On the EC2 home dashboard, look out for any button that says "Launch Instance", and click on it.

![(a post image sharing more information)](/8_img-1.png)

You should be brought to a screen that looks like the one below.

![(a post image sharing more information)](/8_img-2.png)

Now proceed to setup the instance.

### 1. Name and tags.

Simply give your server a name that resonates with your project - I'll call mine "tutorial-instance".

![(a post image sharing more information)](/8_img-3.png)

### 2. Application and OS Images (Amazon Machine Image).

- **OS: Select "Ubuntu"**. 

> Ubuntu operating system(OS), is arguably the most used for virtual machine deployments around the world.
> With much usage by the community, and it's track record of excellent functionality, you're sure to have both reliability,
> and access to community assistance(especially in the form of tutorials and guides) in case anything goes wrong.

- **Amazon Machine Image (AMI)**: Select the latest Ubuntu AMI that is free-tier eligible from the list.

When writing this article, the latest with free-tier support(which I'll be going with), is **Ubuntu Server 24.04 LTS...**.

> If you're building a standard and well-funded project, then feel free do a research, and select a more standard 
> AMI option that matches your project demands from the list.
>
> Note carefully how the various AMIs are for different applications - with a number of them being for AI use-cases.
> Ensure to make the appropriate selection that matches the current project needs.
> 
> Either way, the free-tier AMI should do for quite a lot of applications/use-cases.

- **Architecture**: Ensure that **64-bit x86** is selected.

![(a post image sharing more information)](/8_img-4.png)

### 3. Instance type.

This is a very important section, as it has to do with different important aspects like:

1. The virtual machine specs(CPU) and RAM.
2. The instance cost.

Same as for the AMI, simply go with the highest spec that is free-tier eligible. If you're working on a 
funded project, do your research, and select the perfect fit for the project requirements.

> For this tutorial, I'll go with the free-tier **t3-micro** instance type.

![(a post image sharing more information)](/8_img-5.png)

### 4. Key pair(login).

This section is also a very important one. 

A key-pair is simply what helps to grant you remote SSH access into the server/VM to be created.

> Read this recent Zed Labs Blog Article on 
> [how public key SSH authentication actually works](https://blog.zedlabs.xyz/cloud-and-devops/understanding-how-public-key-ssh-authentication-actually-works),
> to learn and understand what happens under the hood when you attempt an SSH connection with the key pair that will be created.

Now back to the main talk.

Select **Create new key pair** just beside the drop-down menu. 

![(a post image sharing more information)](/8_img-24.png)

Proceed to make the below selections on the resulting pop-up to create you key-pair.

> Feel free to select an older key-pair from the drop-down menu if you or your team already have previously created ones.
> I recommend creating a new one for your current connection. 

- **Key pair name**: simply add any name that fits. Mine is "tutorial-instance-key-pair".

- **Key pair type**: select **ED25519**. 

The image below shed light on why to chose which from the two available options.

![(a post image sharing more information)](/8_img-6.png)

Note that the selection to be made, is affected by both the machine you'll be connecting from, and the VM to be created.
Our instance(VM) to be created, is a modern server, and I believe your local machine is not an 'old-lady' either,
so **ED25519** - since it's more modern and secure, should do.

Feel free to select the other option if you'll be connecting from an older device that may not support the **ED25519** 
cryptographic algorithm.

- **Private key file format**:

1. If you'll be connecting from a shell client that has OpenSSH installed: use **.pem**.
2. If you'll be connecting from PuTTY: use **.ppk**.

> Select **.pem** since that matches the environment that I'll be connecting from in this guide.
> Your code editor terminal and other modern shell interfaces, should work perfectly with **.pem** as well.

Next, click the button, and create the key-pair.

![(a post image sharing more information)](/8_img-7.png)

> The key-pair("tutorial-instance-key-pair.pem" in my case) will be created, and will be downloaded to my computer.

### 5. Network settings.

For the network settings, we'll leave everything as it is by default. We'll learn more about some or
all of the default selections(security groups, VPC, Subnets, and more) in subsequent Zed Labs blog articles.

![(a post image sharing more information)](/8_img-8.png)

### 6. Configure storage.

For storage, simply update the pre-inserted space amount from **8** to **30**.

![(a post image sharing more information)](/8_img-10.png)

----

And that will be it for all the necessary selections.

View your instance summary on the right side of the screen, and without any further selection, proceed to click **"Launch instance"**.

![(a post image sharing more information)](/8_img-9.png)

Once your instance is successfully created, you'll be directed to a this next screen. Scroll to the button of the page, and click on
**"View all instances"**.

![(a post image sharing more information)](/8_img-11.png)

![(a post image sharing more information)](/8_img-14.png)

- Click the checkbox next to your newly created instance, to pre-view it's details.

![(a post image sharing more information)](/8_img-15.png)

- Click on the "Instance Id" and proceed to the instance page proper.

![(a post image sharing more information)](/8_img-16.png)

## Connecting To The Newly Created EC2 Instance.

On the instance page, Click the "Connect" button to head to the connection screen.

![(a post image sharing more information)](/8_img-17.png)

I have two recommendations, with respect to connecting to your EC2 instances

1. **EC2 Instance Connect**. 

This default selection helps you connect to your instance directly from inside the browser.
With this, AWS simply provide you an in-browser shell interface from which you can perform actions
on the newly created virtual machine.

While on it(EC2 Instance Connect), click the "Connect" button to see this in action.

![(a post image sharing more information)](/8_img-17.png)

In the resulting shell interface, type the below command and press "Enter" to see the current VM user.

<HighlighterWrapper
language="bash" 
content={`whoami`}/>

You should get the response("ubuntu") as shown in the image below.

![(a post image sharing more information)](/8_img-18.png)

> From this in-browser browser shell, you can conveniently use your VM as desired. But due to performance/speed issues
> and other constraints. I recommend using the **"SSH Client"** option.

2. **SSH Client**.

For the "SSH Client" option, first start by locating the "key-pair" file that was downloaded to your computer earlier, and write out
the path to the file.

![(a post image sharing more information)](/8_img-19.png)

> I prefer saving such inside an "SSH" directory somewhere on my PC.
>
> The path on my machine looks like this **/c/directory-name/another-directory-name/SSH/tutorial-instance-key-pair.pem**.

**Note that if you're working with file paths on a windows operating system environment, you'll need to add
the file paths within quotes when using them in the commands.**

I.e. **"C:\directory-name\another-directory-name\SSH\tutorial-instance-key-pair.pem"** instead of just 
**C:\directory-name\another-directory-name\SSH\tutorial-instance-key-pair.pem**

Now open a new terminal window inside your code-editor, or any other shell environment that supports SSH, 
then do the following - as can be seen on the "SSH Client" tab screen.

- Run the below command to change the permission on the file so that **only** the owner(your current 
computer user) can read it.

<HighlighterWrapper
language="bash" 
content={`chmod 400 <path to SSH key-pair>`}/>

For me, that would be.

<HighlighterWrapper
language="bash" 
content={`chmod 400 /c/directory-name/another-directory-name/SSH/tutorial-instance-key-pair.pem`}/>

- Secondly, run the below command to log into your VM via SSH.

<HighlighterWrapper
language="bash" 
content={`ssh -i <path to SSH key-pair> ubuntu@<public IP or Public DNS of the instance>`}/>


> The public DNS can be found on the fourth todo of the current("SSH Client") option screen.

![(a post image sharing more information)](/8_img-23.png)

In my case, the resulting command will be:

<HighlighterWrapper
language="bash" 
content={`ssh -i /c/directory-name/another-directory-name/SSH/tutorial-instance-key-pair.pem ubuntu@ec2-13-60-96-165.eu-north-1.compute.amazonaws.com`}/>

> This Zed Labs Blog Article on 
> [how public key SSH authentication actually works](https://blog.zedlabs.xyz/cloud-and-devops/understanding-how-public-key-ssh-authentication-actually-works),
> is an awesome resource that explains what the above command is all about, and how it works under the hood.

You'll get a prompt that looks like the one in the screenshot below, type in **yes** and hit "Enter".

![(a post image sharing more information)](/8_img-20.png)

And just like that, you'll be logged into your EC2 virtual machine shell interface.

![(a post image sharing more information)](/8_img-21.png)

Once again, type in:

<HighlighterWrapper
language="bash" 
content={`whoami`}/>

You should get the response("ubuntu") as shown in the image below.

![(a post image sharing more information)](/8_img-22.png)

Congratulations, you just successfully created, and connected to a virtual machine thanks to AWS EC2.
